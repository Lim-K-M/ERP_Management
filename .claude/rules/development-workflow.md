# 개발 워크플로우 규칙 (always-on)

> 이 파일은 `CLAUDE.md`가 `@import`로 끌어와 **매 세션 항상** 적용된다.
> 철학·전체 흐름·실전 예시는 `docs/guides/development-process.md`(사람용 정본)를 근거로 한다.
> 여기에는 **AI가 모든 작업에서 지켜야 할 제약과 절차**만 간결히 적는다.

---

## 핵심 원칙

> **기획은 이미 완료되어 있다. 우리는 구현만 한다.**

기획자가 준 설계서·UI·validation 규칙을 **그대로** 구현한다. 임의로 기능을 추가·변경하지 않는다. **코드를 쓰기 전에 방향을 문서로 합의**하고, 승인 후에만 구현한다(플랜-퍼스트).

---

## 절대 하지 않는다 (금지)

- ❌ 기획 자료 없이 **추측으로 구현 시작**
- ❌ 기획서/`docs/specs/`에 **없는 기능을 임의 추가**
- ❌ **main 브랜치에 직접 커밋**
- ❌ **사용자 승인 없이** 구현 계획 변경
- ❌ 변경 대상이 아닌 **기존 주석·docstring 삭제/수정**

---

## 7단계 절차

작업을 시작할 때 아래 순서를 따른다. 각 단계의 게이트(✋)를 통과하기 전에 다음 단계로 넘어가지 않는다.

### ① 기획 자료 수집
구현 전 아래를 **먼저 확보**한다. 부족하면 **구현을 시작하지 말고 무엇이 부족한지 구체적으로 질문**한다.
- [ ] 기획서/설계서
- [ ] UI 기획안/화면 이미지
- [ ] Validation 규칙(필드별 검증·에러 메시지)
- [ ] 상태 정의(상태값·전환 조건·배지)
- [ ] API 스펙(없으면 Mock 처리)

### ② 구현 계획 수립
1. 수집 자료를 분석하고 **기존 코드와 대조**한다.
2. 계획서를 `docs/plans/YYYY-MM-DD-<주제>.md`에 작성한다(날짜 프리픽스 필수). → `writing-plans` 스킬을 사용한다.
3. 계획서에는 **변경 범위·대상 파일·검증 방법**을 명시한다.
- 아이디어 단계부터 설계가 필요하면 먼저 `brainstorming` 스킬을 쓴다.

### ③ 승인 ✋
- **승인 전에는 소스를 건드리지 않는다.** 명시적으로 멈추고 사용자 승인을 기다린다.
- 계획이 바뀌면 **다시 승인**을 받는다.

### ④ 브랜치 생성
- `feature/<파트명>` 독립 브랜치에서 작업한다. (예: `feature/sim-lab-setup`)
- **main에 직접 커밋하지 않는다.**

### ⑤ 구현
- 커밋은 **의미 단위**로 나눈다.
- 기존 주석·docstring은 변경 대상이 아니면 **보존**한다.
- 기존 기능에 영향을 주는 변경은 **회귀 확인**을 동반한다.
- 영역별 도메인 표준 스킬(`fastapi-service-architecture`/`db-development-postgres`/`jinja2-ssr-frontend`/`docker-compose-server-ops`(미사용/미작성 — 이 프로젝트는 배포를 하지 않아 실제로 켜진 적 없음, `.claude/skills/README.md` §4·부록 참고))을 따른다.
- 계획서대로 실행할 때는 `executing-plans` 스킬을 사용한다.

### ⑥ 검증 ✋
- 브라우저에서 동작을 확인한다.
- 기획서 체크리스트와 대조한다.
- 완료 후 walkthrough를 정리한다.

### ⑦ PR / merge
- PR을 생성한다. UI 변경이 있으면 **스크린샷 첨부를 권장**한다(필수 아님).
- 리뷰 후 merge하고 브랜치를 정리한다. (형식·리뷰 기준은 아래 「협업 규약」을 따른다.)

---

## 협업 규약 (Git · 커밋 · PR · 리뷰)

여러 사람이 각자 AI로 작업해도 산출물이 한 방향으로 수렴하도록 하는 공통 규약. **혼자 작업해도 동일하게 따른다.**

### 브랜치
- `feature/<파트명>`에서만 작업한다. `main`은 PR로만 반영하고 **직접 push 금지**.
- 한 브랜치 = 한 논리적 변경. 오래 끌지 말고 작게 끊어 머지한다.

### 커밋 메시지 (Conventional Commits)
- 형식: `<type>(<scope>): <요약>` — 예: `feat(order): 주문 생성 API 추가`
- `type`: `feat` / `fix` / `refactor` / `docs` / `test` / `chore` / `style` / `perf`
- **의미 단위**로 커밋한다. 요약은 명령형 한 줄, 본문에는 **"왜"** 를 적는다.

### PR
- **작게** 올린다(리뷰 가능한 단위). 거대한 PR 금지.
- 설명에 **무엇을·왜·검증법** + 관련 계획서(`docs/plans/…`)·스펙 링크를 넣는다. UI 변경 시 스크린샷(권장).
- **머지 전 최소 1인 리뷰**(또는 AI 1차 리뷰 → 사람 확인). 리뷰 기준은 **`code-review-standard` 스킬**을 따른다.
- 머지 후 브랜치를 정리한다.

## 문서 체계 (어디에 무엇이 있나)

| 폴더 | 용도 |
|------|------|
| `docs/specs/` | 기획·분석 스펙 — **구현의 기준** |
| `docs/plans/` | 구현 계획서 (날짜 프리픽스) |
| `docs/designs/` | 디자인 양식·작성 안내 |
| `docs/guides/` | 아키텍처·개발자 가이드 (`development-process.md` 위치) |
| `docs/internal/` | 내부 분석·방법론 장기 보관 |

> 구현 판단이 애매하면 추측하지 말고 **`docs/specs/`를 근거로 삼거나 질문**한다.
