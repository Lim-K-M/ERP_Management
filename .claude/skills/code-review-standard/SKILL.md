# 코드리뷰 표준 (Code Review Standard)

협업에서 머지 품질을 일정하게 유지하기 위한 **공통 리뷰 기준**이다. 리뷰어마다(=AI마다) 보는 게 달라지지 않게 한다.

## 경계 (담당 / 위임)
- **담당**: PR/diff 리뷰 시 "무엇을 보나"(체크리스트), 심각도 등급, 피드백 방식.
- **위임**: 브랜치·커밋·PR **형식** 규약은 `development-workflow`(규칙). 각 영역의 "옳은 코드" 기준은 해당 도메인 스킬(①~④)이 진실.

> **사용 흐름**: PR이 올라오면 이 표준으로 **AI가 1차 리뷰** → 사람은 판단·맥락이 필요한 부분에 집중. 빌트인 `/code-review`를 쓸 때도 이 기준에 맞춘다.

---

## 무엇을 보나 (체크리스트)

순서대로 본다. 위쪽이 더 중요하다.

### 1. 프로세스 정합성 (먼저 확인)
- [ ] **승인된 계획 범위 안인가** — `docs/plans/…`·`docs/specs/` 대비 **스코프 크립**이 없는가. 기획에 없는 기능이 슬쩍 들어오지 않았나.
- [ ] 커밋이 의미 단위이고 메시지가 규약(Conventional Commits)을 따르는가.
- [ ] PR이 리뷰 가능한 크기인가.

### 2. 정확성 (Correctness)
- [ ] 로직이 의도대로 동작하는가. 엣지 케이스(빈 값/널/경계/동시성)를 처리하는가.
- [ ] 에러 처리: `try-catch` → `next(err)` → 전역 핸들러 흐름을 지키는가. 삼켜진 에러는 없는가.
- [ ] 멱등성: 중복 요청/유니크 충돌이 안전하게 처리되는가.

### 3. 보안 (Security)
- [ ] 입력 검증(Zod 등)이 Controller에 있는가. 사용자 입력이 검증 없이 흐르지 않는가.
- [ ] **하드코딩된 비밀/ID/만료시간**이 없는가(→ `.env`/설정).
- [ ] (프론트) 동적 출력이 `escapeHtml`을 거치는가(XSS).
- [ ] 권한 체크가 필요한 엔드포인트에 빠지지 않았는가.

### 4. 프로젝트 컨벤션 준수 (도메인 스킬이 진실)
변경 영역에 해당하는 스킬 기준으로 본다:
- [ ] **백엔드**: Router(경로/파싱만)→Service(순수 로직) 계층 분리, 화면 라우터와 API 라우터가 동일 Service 함수 공유, `HTTPException`으로 통일된 에러 응답(422/404/409) — `fastapi-service-architecture`
- [ ] **데이터**: `ddl/`이 SSOT이며 앱은 `MetaData().reflect()`로 읽기만 함(앱이 새 DB 객체를 만들지 않음), 상태 전이는 `ALLOWED_TRANSITIONS` 단일 소스 준수 — `fastapi-service-architecture`
- [ ] **프론트**: 템플릿 상속(`{% extends %}`/`{% block %}`), PRG 패턴(POST→303), `status_badge` 필터 사용, autoescape 유지(고정값 외 `Markup`/`safe` 남용 금지) — `jinja2-ssr-frontend`
- [ ] **배포/운영**: 이미지 태그 핀(`:latest` 금지), 자격증명 `.env` — `docker-compose-server-ops`(미사용/미작성 — 이 프로젝트는 배포하지 않아 실제로 해당 없음, `.claude/skills/README.md` §4·부록 참고)

### 5. 테스트
- [ ] 크리티컬 로직(결제·권한·제어 등)에 단위 테스트가 있는가.
- [ ] 테스트가 의미 있는 케이스를 검증하는가(통과를 위한 빈 테스트가 아닌가).

### 6. 품질·유지보수
- [ ] 중복/불필요한 복잡도가 없는가(DRY, YAGNI).
- [ ] 변경 대상이 아닌 **기존 주석·docstring을 보존**했는가.
- [ ] 기존 기능에 영향 가는 변경에 **회귀 확인**이 동반됐는가.
- [ ] `console.log` 잔존물·디버그 코드·죽은 코드가 없는가.

## 심각도 등급

피드백마다 등급을 붙여 "무엇이 머지를 막는지" 명확히 한다.

| 등급 | 의미 | 머지 |
|---|---|---|
| 🔴 **Blocker** | 버그·보안 결함·스코프 위반·컨벤션 위반 | **차단** — 고쳐야 머지 |
| 🟡 **Should** | 고치는 게 맞지만 후속 가능 | 합의 후 머지 가능 |
| 🟢 **Nit/제안** | 취향·소소한 개선 | 비차단 |

- 🟢는 "nit:" 으로 표시해 작성자가 무시해도 됨을 분명히 한다.

---

## 피드백 방식
- **사람이 아니라 코드**를 가리킨다. 단정 대신 근거를 댄다("X일 때 Y로 깨질 수 있다").
- 막연한 "이상함" 금지 — **무엇이 왜 문제이고 어떻게 고치면 되는지**까지.
- 잘된 점도 한 줄 남긴다(특히 협업 초반).
- 의견이 갈리면 관련 **스킬/스펙을 근거**로 삼는다. 근거가 없으면 그건 nit이다.

## AI 1차 리뷰 출력 형식
사람이 빠르게 훑도록 등급별로 묶어 보고한다. 각 항목: `파일:라인 — [등급] 문제 — 제안`.
스코프 위반(계획 대비)과 Blocker를 **맨 위**에 둔다.
