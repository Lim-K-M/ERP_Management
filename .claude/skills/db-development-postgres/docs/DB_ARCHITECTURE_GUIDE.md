# 데이터베이스 아키텍처 설계 가이드 (PostgreSQL)

본 문서는 데이터베이스 객체 네이밍, 쿼리 작성 스타일 및 주요 설계 결정사항을 정의합니다.

---

## 1. 네이밍 규칙 (Naming Convention)

### 1.1 일반 원칙
- **대소문자**: 가독성을 위해 SQL 문 작성 시 객체명은 **대문자**, 키워드는 **소문자**로 작성합니다. 
  - *참고: PostgreSQL은 따옴표가 없는 식별자를 소문자로 취급합니다.*
- **구분자**: 스네이크 케이스(`_`) 사용.
- **예약어 금지**: `GROUP`, `ORDER`, `USER`(가급적 피함) 등 예약어 사용 금지.

### 1.2 객체별 접두어

| 객체 유형 | 접두어 | 예시 |
|----------|--------|------|
| 테이블 (Table) | `T_` | `T_USER` |
| 뷰 (View) | `VW_` | `VW_USER_INFO` |
| 함수 (Function) | `F_` | `F_GET_STATS` |
| 프로시저 (Procedure) | `PR_` | `PR_BATCH_JOB` |
| 트리거 (Trigger) | 테이블명 + `_TRG` | `T_USER_TRG` |
| 시퀀스 (Sequence) | `SEQ_` | `SEQ_USER_ID` |
| 인덱스 (Index) | `IDX_` | `IDX_T_USER_LOGIN_ID` |

### 1.3 제약조건 네이밍
- **Primary Key**: `PK_테이블명`
- **Foreign Key**: `FK_테이블명_컬럼명`
- **Unique**: `UNQ_테이블명_컬럼명`

### 1.4 데이터 타입 선택 가이드

| 용도 | 권장 타입 | 비고 |
|------|----------|------|
| 가변 문자열 | `VARCHAR(n)` | 길이 제한이 필요한 경우 |
| 대용량 텍스트 | `TEXT` | 길이 제한 없는 본문/내용 |
| 정수 PK / FK | `BIGINT` | `generated by default as identity`와 함께 사용 |
| 수치(금액 등) | `NUMERIC(p,s)` | 정밀도가 중요한 값 |
| 날짜+시간 | `TIMESTAMPTZ` | 시분초 포함, 타임존 보존 (기본 권장) |
| 날짜만 | `DATE` | 시각 불필요 시 |
| 외부 식별자 | `UUID` | 외부 노출용 고유 키 (`uuidv7()` 권장) |
| 불리언 플래그 | `BOOLEAN` 또는 `CHAR(1)` | 사내 관례상 `Y/N` 플래그는 `CHAR(1)` |

## 2. 쿼리 작성 요령 (SQL Style Guide)

### 2.1 DDL 예시
```sql
create table T_USER (
    USER_ID         bigint generated by default as identity,
    LOGIN_ID        varchar(100)    not null,
    USERNAME        varchar(100),
    PASSWORD        varchar(256)    not null,
    ROLE            varchar(10)     default 'USER' not null,
    LOCK_YN         char(1)         default 'N' not null,
    USER_UUID       uuid            not null default uuidv7(),
    CREATED_ON      timestamptz     not null default current_timestamp,
    CREATED_BY      bigint,
    UPDATED_ON      timestamptz,
    UPDATED_BY      bigint,
    --
    constraint PK_T_USER primary key (USER_ID),
    constraint UNQ_T_USER_LOGIN_ID unique (LOGIN_ID),
    constraint UNQ_T_USER_UUID unique (USER_UUID),
    constraint CK_T_USER_LOCK_YN check (LOCK_YN in ('Y', 'N'))
);
```
> **UUID 생성 함수 (버전별)**
> - **PG 18+(권장)**: `uuidv7()` — 시간순 정렬형으로 인덱스 단편화가 적음. 완전 랜덤이 필요하면 `uuidv4()`.
> - **PG 13~17**: `gen_random_uuid()` (코어 내장, 확장 불필요).
> - **PG 12 이하**: `create extension if not exists pgcrypto;` 후 `gen_random_uuid()`.

#### DML 예시 (SELECT 문)
```sql
select U.USER_ID,
       U.USERNAME,
       C.COMPANY_NAME
  from T_USER U
  join T_COMPANY C
    on U.COMPANY_ID = C.COMPANY_ID
 where U.LOCK_YN = 'N'
 order by U.CREATED_ON desc;
```

### 3. 설계 결정사항 (Design Decisions)

### 3.1 감사(Audit) 트리거
PostgreSQL에서는 트리거가 발생할 때 실행될 함수를 먼저 정의해야 합니다. F_AUDIT_LOG()와 같은 공통 함수를 만들어 여러 테이블에 재사용하는 방식을 권장합니다.

### 3.2 함수의 논리적 그룹화
관련 함수들을 묶어 관리할 때는 다음 방식을 사용합니다.
- 하나의 `Schema`로 분리 (예: `auth.F_LOGIN()`),
- 또는 함수명 접두어를 통한 그룹화 (예: `F_USER_GET_LIST`, `F_USER_UPDATE`).

### 3.3 시퀀스 및 PK
`generated by default as identity` 를 사용하여 표준 SQL 방식을 따릅니다. 이는 내부적으로 시퀀스를 생성하고 관리합니다.

## 5. ORM 연동 (Prisma) — SQL이 진실의 원천

ORM은 Prisma를 사용하되, **스키마 정의는 위 SQL(DDL)이 단독 원천**이다. Prisma는 SQL을 인트로스펙트해 얻은 타입드 클라이언트로만 쓴다.

### 5.1 워크플로우 (SQL-first)
```bash
# 1) DDL/함수/트리거를 SQL로 작성하고 DB에 반영한 뒤
psql "$DATABASE_URL" -f ddl/T_USER.sql -f functions/F_AUDIT_LOG.sql

# 2) DB 구조를 schema.prisma로 끌어온다 (migrate 아님)
npx prisma db pull

# 3) 타입드 클라이언트 생성
npx prisma generate
```
> ⚠️ `prisma migrate`는 사용하지 않는다. 스키마를 Prisma가 생성하면 트리거·`uuidv7()`·대문자 네이밍이 깨진다.

### 5.2 인트로스펙트 결과 매핑 예시
`db pull`은 물리 네이밍을 보존하기 위해 `@@map`/`@map`을 자동으로 붙인다. 떼지 말 것.
```prisma
model T_USER {
  USER_ID    BigInt    @id @default(autoincrement()) @map("USER_ID")
  LOGIN_ID   String    @unique @map("LOGIN_ID") @db.VarChar(100)
  USER_UUID  String    @unique @default(dbgenerated("uuidv7()")) @db.Uuid @map("USER_UUID")
  CREATED_ON DateTime  @default(now()) @map("CREATED_ON") @db.Timestamptz()
  CREATED_BY BigInt?   @map("CREATED_BY")
  UPDATED_ON DateTime? @map("UPDATED_ON") @db.Timestamptz()
  UPDATED_BY BigInt?   @map("UPDATED_BY")

  @@map("T_USER")
}
```

### 5.3 감사 컬럼 책임 분리
- **`CREATED_ON` / `UPDATED_ON`**: DB의 `F_AUDIT_LOG()` 트리거가 채운다. Prisma에서 값을 보내지 않는다.
- **`CREATED_BY` / `UPDATED_BY`**: 트리거가 알 수 없으므로 **Prisma Client Extension**에서 인증 컨텍스트의 사용자 ID를 주입한다.

```typescript
// 인증 미들웨어에서 채운 현재 사용자 ID를 write 작업에 자동 주입
const prisma = basePrisma.$extends({
  query: {
    $allModels: {
      async create({ args, query }) {
        args.data = { ...args.data, CREATED_BY: getCurrentUserId() };
        return query(args);
      },
      async update({ args, query }) {
        args.data = { ...args.data, UPDATED_BY: getCurrentUserId() };
        return query(args);
      },
    },
  },
});
```
> 시스템 작업은 예약된 시스템 계정의 `USER_ID`(또는 `null`)를 사용한다.

### 5.4 Repository 계층
ORM이 Data Access Object 역할을 하므로 **별도 Repository 계층을 두지 않고 Service에서 ORM 클라이언트를 직접 호출**한다. (`fastapi-service-architecture` §2)

> 이 섹션(5.1~5.4)은 Prisma Client Extension 예시로 작성되어 있다. 이 프로젝트(ERP_Management)는 Prisma 대신 **SQLAlchemy Core 리플렉션**을 쓰므로, `_BY` 주입 등 ORM 레이어 훅이 필요하면 Service 함수 내에서 직접 처리한다(Prisma Client Extension 같은 미들웨어 훅 메커니즘은 없음). 이 프로젝트는 인증이 없어 `_BY`를 NULL로 유지하므로 해당 없음.
