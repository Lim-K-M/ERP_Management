02

왜 룰을 도입했나 — 말에서 문서로

처음엔 스킬만으로 개발했다. 그런데 "말로 시키는" 방식엔 모두가 볼 기준이 없어서, 같은 기획도 결과가 제각각 발산했다. 그래서 소통을 문서로 옮겼다 — 그게 룰이다.

BEFORE · 말로만 개발

기준이 없어 결과가 발산

  • ✕기획에 없는 기능이 슬쩍 추가된다 — 스코프 크립
  • ✕main에 직접 커밋된다 — 합의·검토 게이트가 없음
  • ✕같은 기획인데 사람마다 결과가 다르다 — 해석이 제각각
뿌리 원인 — "말"엔 모두가 바라볼 단일 기준(SSOT)이 없다.
→
AFTER · 문서 기반 개발 = 룰

문서를 기준으로, 그 위에서 구현

  • ✓기획은 docs/specs/ — 구현의 유일한 기준
  • ✓방향은 docs/plans/로 합의 → 승인 ✋ → 구현
  • ✓main은 PR로만 · 스코프는 스펙 안에서
결과 — 사람·세션이 바뀌어도 같은 문서 = 같은 결과로 수렴.
핵심 말로 시키면 사람마다·세션마다 결과가 달라진다. 문서를 기준으로 합의하고, 그 문서 위에서 AI가 구현하게 만든 것 — 그게 룰이다.
03

규칙(Rule) vs 스킬(Skill)

가장 먼저 잡아야 할 개념. 어떻게 개발하느냐(일하는 방식)는 규칙, 어떻게 구현하느냐(만드는 방법)는 스킬이다. 규칙은 항상 켜져 있고, 스킬은 필요할 때만 불려온다.

04

플랜-퍼스트 7단계 절차

규칙의 핵심. 코드를 쓰기 전에 방향을 문서로 합의하고, 승인(✋) 후에만 구현한다. 게이트를 통과하기 전에는 다음 단계로 넘어가지 않는다.

1

기획 자료 수집

설계서·UI·validation·상태정의·API 스펙 확보. 부족하면 추측 금지, 질문.

2

구현 계획 수립

기존 코드와 대조 → writing-plans로 docs/plans/YYYY-MM-DD-주제.md 작성.

3

승인 GATE ✋

승인 전에는 소스를 건드리지 않는다. 계획이 바뀌면 다시 승인.

4

브랜치 생성

feature/<파트명> 독립 브랜치. main 직접 커밋 금지.

5

구현

의미 단위 커밋 · 기존 주석 보존 · 회귀 확인. 도메인 스킬 + executing-plans.

6

검증 GATE ✋

브라우저 동작 확인 + 기획서 체크리스트 대조 → walkthrough 정리.

7

PR / merge

작은 PR + 무엇·왜·검증법. code-review-standard로 리뷰 후 머지.

↻

혼자 작업해도 동일

여러 사람이 각자 AI로 일해도 산출물이 한 방향으로 수렴하게 만드는 공통 규약.

절대 하지 않는다 · FORBIDDEN
  • 기획 자료 없이 추측으로 구현 시작
  • 스펙에 없는 기능을 임의 추가
  • main 브랜치에 직접 커밋
  • 사용자 승인 없이 계획 변경
  • 변경 대상이 아닌 기존 주석·docstring 삭제

스킬 카탈로그

인프라·DB·프론트·백엔드처럼 만드는 대상마다 도메인 스킬이 있다. 핵심은 하나뿐 — "언제 어떤 스킬을 부르는가". 트리거 문장만 알면 된다.

① 백엔드

부르는 말"이 PR 리뷰해줘"

부르는 말"이거 계획서부터 써줘"

07

스킬 하나 열어보기 — DB(PostgreSQL)

스킬은 "이 도메인은 이렇게 만든다"를 담은 강제 가능한 표준이다. DB 스킬이 무엇을 강제하는지 보면, 스킬 한 장이 어떻게 팀의 일관성을 만드는지 감이 온다.

네이밍 객체마다 접두어 고정

T_테이블VW_뷰F_함수 PR_프로시저IDX_인덱스_TRG 트리거 PK_/FK_/UNQ_/CK_
  • 키워드는 소문자(create/select), 객체·컬럼명은 대문자(T_USER)
  • 파라미터 p_, 지역변수 l_

공통 컬럼 모든 테이블 필수 4개

  • CREATED_ON · CREATED_BY
  • UPDATED_ON · UPDATED_BY
  • _ON은 트리거가, _BY는 앱 레이어(인증 컨텍스트)가 채운다

UUID 외부 노출 식별자

  • PG18+ uuidv7() · PG13~17 gen_random_uuid()
  • 시간순 정렬 → 인덱스 단편화 적음
-- 공통 감사 트리거 함수 (모든 테이블 재사용)
create or replace function F_AUDIT_LOG()
returns trigger as $$
declare l_user_id bigint;
begin
    l_user_id := cast(current_setting('app.current_user_id', true) as bigint);
    if (tg_op = 'INSERT') then
        new.CREATED_ON := current_timestamp;
        new.CREATED_BY := coalesce(new.CREATED_BY, l_user_id);
    elsif (tg_op = 'UPDATE') then
        new.UPDATED_ON := current_timestamp;
        new.UPDATED_BY := coalesce(new.UPDATED_BY, l_user_id);
    end if;
    return new;
end; $$ language plpgsql;

-- 테이블에 적용 (트리거명 = 테이블명 + _TRG)
create trigger T_ORDER_TRG before insert or update on T_ORDER
    for each row execute function F_AUDIT_LOG();

Prisma 연동 SQL이 진실의 원천 (SSOT)

  • 스키마는 SQL로 작성 → prisma db pull로 인트로스펙트. prisma migrate 금지.
  • 인트로스펙트된 @@map("T_ORDER") 네이밍을 떼지 않는다.
  • 트리거·uuidv7() 기본값은 SQL 소관 — Prisma가 관리하지 않는다.
  • Repository 계층 없이 Service에서 Prisma Client 직접 호출.

01

어디에 두나 & 언제 로드되나

VSCode에서 프로젝트 폴더를 열면 Claude Code가 워크스페이스 루트의 CLAUDE.md와 .claude/를 자동으로 인식한다. 파일을 올바른 위치에 두는 것이 전부다.

프로젝트 루트/ ├─ CLAUDE.md # 프로젝트 헌법 · 항상 로드 │ └ @.claude/rules/... 를 import ├─ .claude/ │ ├─ rules/ │ │ └ development-workflow.md # always-on │ └─ skills/ │ ├─ frontend-ssr/ │ │ └ SKILL.md # 트리거 시 │ ├─ db-development-postgres/ │ │ └ SKILL.md │ └─ .../ ├─ docs/ # specs · plans · guides └─ client/ · server/ # 실제 코드
항상

CLAUDE.md

루트에 두면 매 세션 자동 로드. 프로젝트 개요·코드 스타일·작업 규칙의 단일 기준점. 새 세션·새 멤버가 이것만 읽으면 컨벤션 파악.

@import · 항상

.claude/rules/

CLAUDE.md가 @.claude/rules/development-workflow.md로 끌어온다. 그래서 규칙은 늘 켜져 있다. 승인 게이트·main 커밋 금지 같은 가드레일.

트리거 시만

.claude/skills/<이름>/SKILL.md

평소엔 이름과 한 줄 설명만 로드. 작업이 그 설명과 맞거나 /스킬명을 치면 그때 본문 전체를 불러온다 → 컨텍스트 절약.

02

VSCode에서 매일 쓰는 4가지 방법

사이드바의 Claude Code 패널에서 대화한다. 대부분은 그냥 자연어로 말하면 되고, 나머지는 기호 4개로 제어한다.

자연어

말하면 스킬이 자동 발동

SKILL.md의 description과 요청이 매칭되면 해당 스킬이 자동 로드된다. 스킬을 외울 필요 없이 하려는 일을 말한다.

"이력 테이블 만들어줘" → db 스킬

04

초기 환경 세팅 — PostgreSQL

코드는 스킬이 짜주지만 환경은 사람이 먼저 깔아둬야 한다. PostgreSQL은 그냥 설치돼 있는 게 아니라 Docker로 띄워야 비로소 존재한다 — 이게 돼야 스킬이든 뭐든 돌릴 수 있다.

1

compose 서비스로 DB 띄우기

이미지 태그는 정확히 핀한다(:latest 금지). 비밀번호·최상위 볼륨 선언이 빠지면 안 뜬다. Windows에 postgres가 이미 5432를 쓰면 호스트 포트를 5433으로 피한다.

# docker-compose.yml
services:
  postgres:
    image: postgres:16.4              # :latest 금지 · 태그 핀
    environment:
      POSTGRES_DB: emko
      POSTGRES_USER: emko
      POSTGRES_PASSWORD: ${DB_PASSWORD}   # .env 주입 · 없으면 안 뜸
    ports: [ "5433:5432" ]        # 호스트5433 → 컨테이너5432 (5432 선점 피함)
    volumes: [ pgdata:/var/lib/postgresql/data ]
volumes:                          # ← 이름 있는 볼륨은 반드시 선언
  pgdata:
$ docker compose up -d postgres
$ docker compose ps   # PORTS: 0.0.0.0:5433->5432 · STATUS 확인
2

.env 로 비밀번호 주입

비밀번호는 하드코딩 금지, 항상 .env. compose의 ${DB_PASSWORD}가 여기서 채워진다. docker-compose.yml과 같은 폴더에 둔다.

# .env
DB_PASSWORD=1234
3

접속 확인

외부(DBeaver·앱)는 localhost:5433 · DB·유저 emko · 비번은 .env 값. 컨테이너 안에서 바로 확인도 된다:

# 컨테이너 내부 포트는 5432 (호스트만 5433)
$ docker compose exec postgres psql "postgresql://emko:1234@localhost:5432/emko" -c "select 1;"
자주 겪는 함정 ① 비번은 볼륨 첫 초기화 때만 굳는다 — 바꾸려면 docker compose down -v로 볼륨을 밀고 다시 up. · ② compose 파일을 고쳐도 down/up 전엔 반영 안 됨. · ③ Windows가 5432를 선점하면 localhost가 컨테이너가 아닌 그쪽으로 간다 → 포트를 5433으로 피하면 깔끔.
여기서부터는 스킬DB가 뜨면 테이블·함수·트리거 같은 DB 객체는 db-development-postgres 스킬이 표준(네이밍·감사컬럼·트리거)대로 만든다. 사람은 환경만 준비하면 된다.
05

DB 접속 도구 — pgAdmin & DBeaver

DB를 띄웠으면 GUI 툴로 붙어서 테이블을 보고 쿼리를 날린다. 둘 다 우리 컨테이너(localhost:5433)에 붙는 클라이언트일 뿐 — DB 자체가 아니다. 하나만 골라 쓰면 된다.

범용 · 무료 · 오픈소스

DBeaver

PostgreSQL·MySQL·Oracle 등 여러 DB를 한 툴로. 평소 여러 DB를 다룬다면 이쪽.

설치 dbeaver.io/download → Community Edition 받기 → 설치
접속 New Connection → PostgreSQL 선택 → 아래 정보 입력 → Test Connection → Finish
비고 데스크탑 전용 — 컨테이너로는 안 뜬다 (웹으로 쓰려면 별도 제품 CloudBeaver)
PostgreSQL 전용 · 무료 · 공식

pgAdmin

PostgreSQL 공식 관리 툴. Postgres만 깊게 쓸 때.

설치 pgadmin.org/download → OS에 맞는 버전 받기 → 설치
접속 Servers 우클릭 → Register → Server → General에 이름, Connection 탭에 아래 정보 → Save
비고 데스크탑 설치 또는 도커로 띄우기 둘 다 가능 ↓
접속 정보 · 데스크탑 앱에서 (호스트에서 붙을 때)
Host
localhost
Port
5433
Database
emko
Username
emko
Password
1234
➕ pgAdmin을 도커로 띄우기 (올-도커 · 팀 공용) 
# docker-compose.yml — services: 아래에 추가
  pgadmin:
    image: dpage/pgadmin4:8.14          # :latest 금지 · 태그 핀
    environment:
      PGADMIN_DEFAULT_EMAIL: admin@emko.com
      PGADMIN_DEFAULT_PASSWORD: ${PGADMIN_PASSWORD}   # .env 에 추가
    ports: [ "5050:80" ]
    depends_on: [ postgres ]
# .env — 비번 두 개 (서로 다름)
DB_PASSWORD=1234              # postgres 접속 비번
PGADMIN_PASSWORD=admin1234    # pgAdmin 로그인 비번
실행 서비스 추가 후 sudo docker compose up -d 다시 — postgres는 그대로, pgAdmin만 새로 뜬다
접속 브라우저 http://localhost:5050 → 로그인(.env의 이메일·비번) → Servers에 서버 등록
⚠ 여기선 접속정보가 다르다 컨테이너 pgAdmin은 postgres와 같은 도커 네트워크 안이라, 서버 등록 시 Host = postgres(서비스명) · Port = 5432(내부 포트). localhost:5433이 아니다!
자주 겪는 함정 ① Port — 데스크탑 앱은 localhost:5433, 컨테이너 pgAdmin은 postgres:5432 (내부). · ② Password는 .env 값. · ③ 안 붙으면 sudo docker compose ps로 STATUS·PORTS부터 확인.

01

과제 개요 & 규칙

각자 설계한 직원(emp) ERD·DDL을 기획 자료로 삼아, 직원관리 웹사이트를 AI 코딩으로 개발한다. 목적은 화려한 결과물이 아니라 — AI 개발에 익숙해지고, 스킬 + 룰(방법론)을 몸으로 익히는 것이다.

기획은 이미 완료되어 있습니다. 여러분은 구현에 집중하면 됩니다 — 요구사항 정의서를 그대로 구현하고, 코드 쓰기 전에 계획서로 합의합니다.
규칙 01

스펙 그대로

아래 요구사항 정의서 범위 안에서 구현한다. 임의 기능 추가 금지. 애매하면 추측 말고 질문.

규칙 02

플랜-퍼스트

docs/plans/에 계획서 작성 → 승인 ✋ 후 구현 시작. 코드부터 짜지 않는다.

규칙 03

main 보호

feature/ 브랜치에서 작업하고 PR로만 반영. main 직접 커밋 금지.

규칙 04

본인 ERD · 로컬 실행

언어·프레임워크는 자유. 데이터는 각자 설계한 ERD를 PostgreSQL에 올려 쓴다. 배포는 하지 않고 로컬에서 실행·확인.

02

요구사항 정의서

"무엇을 만드나"의 기준. 필드·스키마는 각자 설계한 ERD를 따르고, 공통으로 맞추는 건 상태값(재직·휴직·퇴직)뿐이다. 이걸 근거로 계획서를 쓴다.

기능 요구사항

F-01

직원 목록 조회

본인 ERD의 주요 필드와 재직상태를 표로. 상태는 배지로 표시.

F-02

직원 등록

본인 ERD의 필드대로 입력 → 저장. 본인이 정한 validation을 통과해야 저장.

F-03

직원 상세 / 수정

한 직원의 정보를 보고 수정할 수 있다.

F-04

상태 변경

재직 → 휴직 → 복직 → 퇴직. 변경 시 목록의 배지가 즉시 바뀐다.

F-05

검색 · 필터 (선택)

이름·부서·상태로 목록 필터링.

필드 · 검증

각자 설계한 ERD를 따릅니다. 어떤 컬럼을 두고 무엇을 필수·유니크·형식으로 검증할지는 본인 설계대로 — 공통으로 맞추는 건 아래 상태값뿐입니다.

공통 상태 정의 — 모두 동일하게 (재직·휴직·퇴직)

재직정상 근무 중
휴직복직 가능
퇴직종료 상태
전환: 재직 ⇄ 휴직 → 퇴직  ·  코드값은 자유 (예: ACTIVE / LEAVE / RESIGNED)

API 스펙 개요 — REST 예시(스택 무관)

GET/employees목록
POST/employees등록
GET/employees/:id상세
PATCH/employees/:id수정
PATCH/employees/:id/status상태 변경

화면: 목록 페이지 · 등록/수정 폼 · 상세. 데이터는 각자 설계한 ERD/테이블을 PostgreSQL에 올려 사용합니다.

힌트 & 제출
