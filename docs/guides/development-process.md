# 개발 프로세스 가이드

> 이 프로젝트가 **어떻게 굴러가는지**(프로세스 흐름)와 **어떻게 활용하는지**(실전 사용법)를 정리한 문서.
> 새로 합류했거나 오랜만에 작업을 재개할 때 이 문서부터 읽으면 된다.

---

## 1. 핵심 철학

> **기획은 이미 완료되어 있다. 우리는 구현만 한다.**

- 기획자가 제공한 설계서·UI·validation 규칙을 **그대로** 구현하는 것이 목표.
- 임의로 기능을 추가하거나 변경하지 않는다.
- **코드를 쓰기 전에 방향을 문서로 합의**하고, 승인 후에만 구현한다 (플랜-퍼스트).

이 철학의 목적은 소규모 팀에서도 **재작업·스코프 크립·무질서한 main**을 막고, 사람과 AI(Claude Code)가 같은 규칙으로 일관되게 일하도록 만드는 것이다.

---

## 2. 프로세스 전체 흐름

```
① 기획자료 수집 ──▶ ② 계획서 작성 ──▶ ③ 승인 ──▶ ④ 브랜치 생성 ──▶ ⑤ 구현 ──▶ ⑥ 검증 ──▶ ⑦ PR/merge
   (자료 부족 시           (docs/plans/)      (승인 전엔        (feature/<파트명>)   (의미단위      (기획 대조)    (UI면
    여기서 질문)                              코드 X)                            커밋)                       스크린샷)
```

### ① 기획 자료 수집

구현 시작 전, 아래 항목을 **먼저 확보**한다. 부족하면 구현을 시작하지 말고 구체적으로 질문한다.

- [ ] 기획서/설계서 (마크다운, PDF 등)
- [ ] UI 기획안 / 화면 이미지 (레이아웃, 컴포넌트 배치)
- [ ] Validation 규칙 (필드별 검증 조건, 에러 메시지)
- [ ] 상태 정의 (상태값, 전환 조건, 배지 표기)
- [ ] API 스펙 (없으면 Mock으로 처리)

### ② 구현 계획 수립

1. 수집한 자료를 분석해 기존 코드와 대조한다.
2. 구현 계획서를 작성한다 — 파일명에 **작성 날짜를 프리픽스**로 붙인다: `docs/plans/YYYY-MM-DD-<주제>.md` (예: `docs/plans/2026-06-26-sim-lab-setup.md`). 언제 만든 계획서인지 한눈에 보이게 한다.
3. **사용자 승인을 받은 후에만** 코드 작성을 시작한다.

### ③ 승인

- 승인 전에는 소스를 건드리지 않는다.
- 계획이 바뀌면 다시 승인을 받는다 (임의 변경 금지).

### ④ 브랜치 생성

- 각 기능은 **독립 브랜치**에서 개발한다.
- 네이밍: `feature/<파트명>` (예: `feature/sim-lab-setup`, `feature/detail-modal`)
- **main 브랜치에 직접 커밋하지 않는다.**

### ⑤ 구현

- 커밋은 **의미 단위**로 나눈다.
- 기존 주석·docstring은 변경 대상이 아니면 **보존**한다.
- 기존 기능에 영향을 주는 변경은 **회귀 확인**을 동반한다.

### ⑥ 검증 및 완료

- 브라우저에서 동작을 확인한다.
- 기획서 체크리스트와 대조한다.
- 완료 후 walkthrough를 정리한다.

### ⑦ PR / merge

- PR을 생성한다.
- UI 변경이 있으면 **스크린샷 첨부를 권장**한다 (필수 아님).
- 리뷰 후 merge.

## 3. 프로세스를 떠받치는 구성 요소

이 프로세스는 다음 4개의 장치로 강제·기록된다.

### 3.1 `CLAUDE.md` — 프로젝트 헌법

- 프로젝트 개요·아키텍처·코드 스타일·작업 규칙을 한곳에 명시.
- 사람과 Claude Code가 **같은 규칙**으로 동작하게 하는 단일 기준점.
- 새 멤버/새 세션은 이 파일만 읽으면 컨벤션을 파악할 수 있다.

**작업 규칙 요약:**
1. 소스 수정 전 **방향을 먼저 확인**받는다.
2. 기존 기능에 영향을 주는 변경은 **회귀 확인** 필수.
3. `docs/specs/`에 정의된 스펙 범위를 벗어나지 않는다.
4. PR 생성 시 **스크린샷 첨부 권장** (UI 변경 시 — 필수 아님).
5. 기존 주석·docstring은 변경 대상이 아니면 보존한다.

### 3.2 `.claude/rules/development-workflow.md` — 작업 규칙

- 위 1~4단계 프로세스를 규칙으로 명문화한 파일.
- "기획 완료, 구현만" 철학과 금지 사항(아래)을 담는다.

### 3.3 `docs/` — 역할별 문서 체계

문서를 역할별로 나눠 두어 **"이 정보가 어디 있지?"를 없앤다.**

| 폴더 | 용도 |
|------|------|
| `docs/specs/` | 기획·분석 스펙 (시스템 분석, ERD, 시나리오 등) — **구현의 기준** |
| `docs/plans/` | 수행·구현 계획서 (날짜 프리픽스 권장) |
| `docs/designs/` | 디자인 양식·작성 안내 |
| `docs/guides/` | 아키텍처·개발자 가이드 (이 문서가 여기 위치) |
| `docs/internal/` | 내부 분석·방법론 등 장기 보관 자료 |

### 3.4 브랜치 + PR — 안전벨트

- `feature/<파트명>` 브랜치로 작업을 격리한다.
- PR을 통해서만 main에 반영한다 → main은 항상 동작 상태를 유지한다.
- 변경이 리뷰 가능한 단위로 남아 히스토리 추적이 쉽다.

---

## 4. 금지 사항

- ❌ 기획 자료 없이 추측으로 구현 시작
- ❌ 기획서에 없는 기능을 임의 추가
- ❌ main 브랜치에 직접 커밋
- ❌ 사용자 승인 없이 구현 계획 변경

## 5. 실전 사용법 — 한 기능을 처음부터 끝까지

예: "시뮬레이션 랩 설정 화면"을 개발한다고 하자.

```
1. 기획자에게 자료 요청
   → 설계서 + UI 이미지 + validation 규칙 + 상태 정의 (+ API 스펙)
   → 부족하면 "무엇이 부족한지" 구체적으로 질문

2. 계획서 작성
   → docs/plans/2026-05-12-sim-lab-development-plan.md
   → 기존 코드와 대조, 변경 범위·파일·검증 방법 명시

3. 승인 받기
   → 계획서 리뷰 → OK 받기 전엔 코드 X

4. 브랜치
   → git checkout -b feature/sim-lab-setup

5. 구현
   → 의미 단위 커밋, 기존 주석 보존, 회귀 영향 체크

6. 검증
   → 브라우저 확인 + 기획 체크리스트 대조

7. PR
   → 스크린샷(권장) → 리뷰 → merge → 브랜치 정리
```

---

## 6. 활용 팁

- **계획서는 가볍게, 그러나 반드시.** 한두 단락이라도 "무엇을·왜·어떻게·검증법"을 적고 승인받는 습관이 재작업을 막는다.
- **`docs/specs/`를 항상 먼저 본다.** 구현 판단이 애매하면 추측하지 말고 스펙을 근거로 삼거나 질문한다.
- **Claude Code에게 맡길 때도 같은 절차를 따르게 한다.** "방향 먼저 확인 → 승인 → 구현"을 지시하면 AI가 멋대로 범위를 넓히는 것을 막을 수 있다.
- **날짜 프리픽스 문서명**(`YYYY-MM-DD-주제.md`)으로 결정의 타임라인을 남긴다.

## 7. 필요해지면 추가할 것 (지금은 선택)

팀이 커지거나 문제가 반복될 때 골라서 도입한다. 미리 전부 도입하면 소규모 팀엔 부담이 된다.

| 도입 후보 | 추가 시점 |
|---|---|
| `docs/troubleshootings/` | 같은 버그/이슈가 반복될 때 (재발 방지 기록) |
| 단계별 보고서 (`_stage{N}` / `_report`) | 한 작업이 여러 날·여러 단계로 길어질 때 |
| 타스크 번호·마일스톤 체계 | 이슈가 많아져 번호로 추적해야 할 때 |
| `CONTRIBUTING.md` | 외부 기여자가 생길 때 |
