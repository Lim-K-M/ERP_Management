# AI 활용 정리

> 이 문서는 인턴 과제(직원관리 시스템) 진행 과정에서 AI(Claude Code, Sonnet 5)를 어떻게 활용했는지, 무엇이 잘 됐고 무엇을 사람이 바로잡았는지를 정리한다.

## 1. 사용 도구

- **Claude Code (Sonnet 5)** — 코드 작성, 계획서 작성, 브라우저 자동화 검증, git/PR 운영까지 전 과정에 사용.
- 로컬 개발 환경: FastAPI + Jinja2 + PostgreSQL(Docker), Playwright(스크린샷 자동화), GitHub CLI(PR 생성).

## 2. 워크플로우: "기획은 끝났다, 구현만 한다"

`.claude/rules/development-workflow.md`에 정의된 절차(기획자료 수집 → 계획서 → 승인 → 브랜치 → 구현 → 검증 → PR)를 AI가 그대로 따르도록 세션 내내 요구했다. 실제로:

- 요구사항 정의서(`docs/specs/2026-07-08-employee-management-spec.md`)를 먼저 읽고 F-01~F-05 + bonus 범위를 파악한 뒤에만 구현을 시작했다.
- 로그인 기능, 인사발령 이력(bonus), 사번/이메일/전화번호 형식 강화처럼 **스펙에 없거나 스펙을 벗어나는 변경**은 AI가 먼저 "이건 스펙 §7의 '인증 없음'과 충돌한다" 같은 형태로 문제를 짚고, 진행 여부를 사용자에게 직접 물어본 뒤에만 계획서(`docs/plans/`)를 쓰고 구현했다. 임의로 기능을 추가한 적은 없다.
- 크지 않은 변경(사번 자리수 규칙 반영, 팝업 추가 등)은 대화 중 바로 구현했고, 새 스키마·서비스·라우터가 얽히는 굵직한 기능(로그인, 인사발령 이력)은 반드시 계획서 작성 → 승인 절차를 거쳤다.

## 3. AI가 실제로 검증한 방식

"구현했다"는 말을 그대로 믿지 않고 매번 실제 서버(uvicorn) + 실제 PostgreSQL로 검증했다.

- `docker compose up` → DDL 적용 → 서버 기동까지 직접 수행.
- curl로 API를 직접 호출해 상태 코드(200/303/401/404/409/422)를 확인.
- 데이터베이스에 `psql`로 직접 접속해 저장된 값(이력 테이블 순서, 사번 형식 등)을 확인.
- 결과화면 스크린샷은 Playwright로 헤드리스 브라우저를 직접 구동해 로그인 → 등록 → 부서/직급 변경 → 상태변경까지 실제 시나리오를 수행한 뒤 캡처했다(가짜로 화면을 그린 게 아니라 실제 동작 결과).

## 4. AI가 찾아낸 버그 (검증 중 발견)

| 버그 | 발견 계기 |
|---|---|
| `TemplateResponse` 신버전 시그니처 불일치 | 실제 서버 기동 후 브라우저 확인 중 발견 |
| 상세페이지 404 처리 누락 | 존재하지 않는 emp_id로 직접 호출해 발견 |
| 데이터 무결성 갭(순환참조/레이스컨디션/퇴직자매니저지정/사번대소문자/부서필터 빈값) | 별도 무결성 리뷰 요청 시 코드 전체를 다시 훑어 발견 |

## 5. AI의 실수와 사람이 바로잡은 사례

AI가 항상 옳았던 것은 아니다. 이번 과제에서 실제로 발생한 실수:

1. **사번 형식 정규식 오류** — 사용자는 "알파벳 **A** + 숫자 4자리"라고 명확히 요청했는데, AI가 나중에 형식 검증을 추가하며 `^[A-Z]\d{4}$`(임의의 알파벳 한 글자)로 잘못 일반화했다. 이 버그는 AI가 데모 데이터를 만들며 `D0001`을 등록해버려서 **사용자가 직접 발견**했고, `^A\d{4}$`로 수정했다(PR #15).
2. **`main` 브랜치 직접 커밋** — 문서 커밋 하나를 실수로 `main`에 직접 커밋했다. AI가 커밋 직후 스스로 인지하고, 해당 커밋을 별도 브랜치로 옮기고 `main`은 원격 상태로 리셋한 뒤 정상적으로 PR을 통해 반영했다.
3. **한글 인코딩(mojibake) 문제** — 터미널(curl)로 한글이 포함된 요청을 보내는 과정에서 인코딩이 깨져 직원 이름이 `ÀåÀº¿ì` 같은 형태로 저장된 적이 있다. 시딩 공식(이메일/전화번호/부서/직급이 `emp_id` 기준 결정론적으로 생성됨)을 역산해 원래 값을 복구했다.
4. **PR 병합 순서로 인한 스크린샷 불일치** — 스크린샷 PR이 코드 수정 PR보다 먼저 병합되면서, `main`에 구버전 스크린샷이 남는 일이 있었다. 이후 별도 PR로 최신 스크린샷으로 교체했다.

이런 사례들은 "AI가 만든 결과물도 실제로 확인해야 한다"는 걸 그대로 보여준다 — 특히 사용자가 준 요구사항의 **정확한 표현**(예: "A"라는 특정 글자 vs "알파벳"이라는 범주)을 AI가 코드로 옮기는 과정에서 미묘하게 일반화·왜곡할 수 있다는 점.

## 6. 브랜치/PR 운영

과제 규칙(`main` 직접 커밋 금지, `feature/*` 브랜치 + PR)을 그대로 따랐다. 총 16개의 PR이 순차적으로 생성·병합되었으며(`ERP_initial_setup`부터 시작해 F-01~F-05, bonus, 로그인 확장, 버그 수정, 문서화까지), 각 PR은 아래 원칙을 지켰다:

- 한 PR = 한 논리적 변경 단위.
- 커밋 메시지는 Conventional Commits 형식(`feat`/`fix`/`docs`/`chore` 등).
- 스펙 범위를 벗어나는 확장은 PR 설명과 `docs/internal/progress-checklist.md`에 별도로 명시.

## 7. 총평

- **잘 된 점**: 계획서 우선 절차 덕분에 로그인·인사발령 이력처럼 스펙 밖 확장이 여러 개 겹쳐도 무엇이 "기본 스펙"이고 무엇이 "확장"인지 문서상 명확히 구분됐다. 실제 서버/DB로 매번 검증하는 습관 덕분에 여러 버그를 구현 직후 바로 잡을 수 있었다.
- **아쉬운 점**: AI가 사용자의 표현을 코드로 옮길 때(정규식, 문구 등) 미묘하게 일반화하는 실수가 있었다 — 이런 부분은 결과물을 실제로 테스트 데이터로 찔러봐야 드러난다. 브랜치를 병렬로 여러 개 운영하다 보니 PR 병합 순서에 따라 산출물(스크린샷)이 어긋나는 운영상의 허점도 있었다.
