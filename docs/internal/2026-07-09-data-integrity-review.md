# 데이터 무결성 리뷰 — 직원 추가/수정/상태변경 (2026-07-09)

> 직원 등록·수정·상태변경 시 데이터 무결성이 깨질 수 있는 지점을 코드 기준으로 점검한 결과.
> 대상 코드: `app/services/employee_service.py`, `app/schemas/employee.py`, `ddl/tables/T_EMPLOYEE.sql`.

## 전제

**직원 삭제 기능은 아직 없다.** `app/services/employee_service.py`에 delete 계열 함수가 없다(grep으로 확인). 아래는 현재 존재하는 추가/수정/상태변경 경로를 중심으로 점검했고, 삭제 기능을 나중에 추가할 때 특히 주의할 지점도 함께 남겼다.

## 🔴 실제 데이터 무결성에 영향 있음

1. **매니저 순환 참조(cycle) 가능**
   `manager_id`는 "존재하는 직원인지"와 "자기 자신은 아닌지"만 검사한다(`_validate_references`). A의 매니저를 B로, B의 매니저를 다시 A로 지정하는 것이 그대로 통과된다. DB도 이를 막지 못한다(순환 참조는 FK로 표현 불가). 조직도를 순회하는 기능을 나중에 추가하면 무한루프 위험.

2. **상태 변경 동시성(레이스 컨디션)**
   `change_status`는 현재 상태를 `SELECT` → 검증 → `UPDATE` 순으로 처리하는데, 그 사이 락(`SELECT ... FOR UPDATE`)이나 `WHERE emp_status = 현재값` 조건이 없다. 두 요청이 거의 동시에 같은 직원의 상태를 바꾸려 하면 둘 다 같은 "현재 상태"를 보고 각자 유효하다 판단해 통과할 수 있고, 나중에 커밋되는 쪽이 조용히 덮어쓴다. 순차적 전이 보장이 깨질 수 있는 지점.

3. **매니저가 퇴직해도 그대로 매니저로 남음**
   `manager_id` 지정 시 "존재하는 직원"인지만 확인하고 그 직원의 `emp_status`는 보지 않는다. `RESIGNED` 상태인 사람을 신규 직원의 매니저로 지정하거나, 이미 매니저인 사람이 퇴직해도 부하 직원들의 `manager_id`는 그대로 남는다(연쇄 처리 없음).

## 🟡 나중에 기능을 추가할 때 터질 수 있는 부분 (지금은 발현 안 됨)

4. **삭제 기능을 추가하면 FK가 막을 것**
   `ddl/tables/T_EMPLOYEE.sql`을 보면 `manager_id`/`dept_id`/`position_id` 외래키에 `ON DELETE`가 지정 안 되어 있어 기본값(NO ACTION/RESTRICT)이다. 즉 "직원 삭제"를 추가하면, 그 직원이 누군가의 매니저이거나 `t_employment_history`(bonus)에 참조돼 있으면 DB가 삭제를 막고 `IntegrityError`를 던진다. 지금 코드엔 이걸 잡는 처리가 없으므로, 삭제 기능을 넣을 때 반드시 같이 처리해야 한다. 부서/직급 삭제도 동일한 문제.

## 🟢 사소한 것

5. **`emp_no` 대소문자 구분**
   `E001`과 `e001`은 DB 유니크 제약상 서로 다른 값으로 취급돼 둘 다 등록될 수 있다. 실무에서는 같은 사번의 오타로 보일 수 있는 케이스.

6. **잘못된 상태값 문자열의 에러 코드가 부정확**
   `emp_status`에 `"FOO"`처럼 존재하지 않는 값을 보내면 "값이 잘못됨"(422)이 아니라 "그 전이가 허용 안 됨"(409)으로 처리된다. `ALLOWED_TRANSITIONS`에 없는 문자열은 뭘 넣어도 이 경로로 빠지기 때문. 데이터가 잘못 저장되는 문제는 아니지만 에러 의미가 부정확하다.

## ✅ 이미 잘 처리된 부분

- `emp_no` 중복은 사전 체크가 아니라 DB `IntegrityError`를 잡는 방식이라, 동시에 같은 사번으로 등록/수정해도 레이스 컨디션 없이 안전하게 막힌다.
- 상태값은 오직 `change_status`를 통해서만 바뀔 수 있고, 일반 정보수정(`EmployeeUpdate`)에는 `emp_status` 필드 자체가 없어서 상태 전이 검증을 우회할 방법이 없다.
- DB에도 `CK_T_EMPLOYEE_STATUS` CHECK 제약이 있어, 앱 로직이 뚫리더라도 잘못된 상태값이 저장될 수 없는 이중 방어가 되어 있다.

## 우선순위 제안

가장 먼저 고칠 만한 것: **1번(매니저 순환)**, **3번(퇴직자 매니저 지정)**. 둘 다 `_validate_references`에 검사를 추가하는 선에서 해결 가능하며, 스펙(`docs/specs/2026-07-08-employee-management-spec.md`)에 명시된 요구사항은 아니므로 착수 전 범위를 다시 확인·승인받는다.
